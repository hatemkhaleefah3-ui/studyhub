PROJECT: Student Study Tracker Website — Feature Fixes & Redesign
CONTEXT: This is a live, production website (deployed via GitHub → Cloudflare) 
used by a student to manage subjects, schedules, checklists, and study 
materials. Reliability and data integrity are the top priority — this is 
an active daily-use tool, not a prototype.

ENTITY TYPES IN THIS APP (for clarity throughout this prompt):
- SUBJECT: an academic subject (e.g., "Math", "Physics"). Has lectures, 
  exams, and study materials.
- FOLDER: a general-purpose container the user creates (not academic-specific).
- TASK: a single checklist or schedule item.
Rules below apply to a specific entity type unless stated as "all items."

════════════════════════════════════════
GLOBAL RULES (APPLY TO EVERYTHING BELOW)
════════════════════════════════════════
1. Do NOT change the site's overall theme, color palette, fonts, spacing 
   system, or navigation structure unless a task explicitly says so.
2. Do NOT refactor, rename, or restructure files/routes/components unrelated 
   to the specific task at hand.
3. Do NOT modify build config, dependencies, or deployment settings.
4. This site is LIVE — every change must be backward-compatible with 
   existing saved data. No data loss on deploy.
5. Maintain full mobile responsiveness for every change — this is used 
   primarily on mobile.
6. Build in this order to minimize risk: 
   Phase 1 (low-risk UI fixes) → Phase 2 (interaction model changes) → 
   Phase 3 (new features/redesigns) → Phase 4 (calendar system, built 
   and tested in complete isolation from the rest).
7. If any requirement conflicts with the current architecture, flag it 
   before implementing rather than making a unilateral structural change.

════════════════════════════════════════
PHASE 1 — LOW-RISK FIXES
════════════════════════════════════════

1.1 FLOATING "+" BUTTON — REMOVE ENTRY ANIMATION
Problem: The "+" button appears center-screen on load, then animates to 
its fixed bottom-right position.
Fix: Render it directly in its final fixed bottom-right position on first 
paint. No entry animation, no layout shift.

1.2 SCHEDULE "SELECTED DAY" — FIX VISUAL OVERLAP
Problem: When a day is selected in the day strip (including auto-selected 
"today"), its scale/pop animation overlaps and clips into the schedule 
bar above it.
Fix: Adjust scale amount, spacing, and stacking order so the highlight 
animation stays fully contained within its own row on mobile and desktop.

1.3 REMOVE "DATA" SECTION FROM SETTINGS
Fix: Remove the "Data" section and its UI from Settings only. Do not 
remove underlying data logic that other features rely on.

1.4 MOBILE HEADER — MOVE ACTIVE COUNTER BESIDE GREETING
Problem: On mobile, the active counter sits below the "Hello, [Name]" 
greeting.
Fix: On mobile viewports only, place it in the same row as the greeting. 
Leave desktop layout unchanged.

════════════════════════════════════════
PHASE 2 — INTERACTION MODEL CHANGES
════════════════════════════════════════

2.1 SWIPE GESTURES FOR EDIT/DELETE (all items: subjects, folders, tasks, 
    checklist items, schedule items)
Problems today: Schedule items can't be edited/deleted at all; checklist 
edit/delete buttons only show when a task is checked.
Fix: Replace static buttons with swipe gestures:
   - Swipe LEFT → RIGHT: Edit — opens the redesigned Add/Edit window 
     (3.3), pre-filled with current data.
   - Swipe RIGHT → LEFT: Delete — moves the item to the Archive (2.2), 
     not permanent deletion.
   - Show a visual cue mid-swipe (background/icon reveal) before the 
     action commits; releasing back to center cancels it.
   - Provide a working non-swipe fallback for desktop/mouse users (e.g., 
     hover-reveal icons or a "..." menu).
Acceptance: Works on all item types regardless of state; doesn't conflict 
with normal scrolling; desktop has a full equivalent.

2.2 ARCHIVE SYSTEM FOR DELETED ITEMS
Fix: Deleted items move to an Archive instead of being permanently removed.
   - Accessible from Settings.
   - Shows what each item was, when it was deleted, and its original 
     category (Schedule / Checklist / Subject / Folder).
   - From Archive: Restore (returns item to original location with all 
     data intact) or Permanently Delete (irreversible, requires confirmation).
   - Archived items do not appear in normal views until restored.

════════════════════════════════════════
PHASE 3 — NEW FEATURES & REDESIGNS
════════════════════════════════════════

3.1 AUTO-ASSIGNED WALLPAPER & COLOR (all items: subjects, folders, tasks)
Fix: On creation, automatically assign a random color and wallpaper from 
a predefined set — no manual selection step anywhere in the Add window.
Acceptance: No item creation flow ever asks the user to pick a color or 
image; each new item is visually distinct immediately.

3.2 PER-ITEM COLOR THEMING (subjects and folders only, scoped per-page)
Fix: A subject's or folder's own page (and its sub-pages) uses its 
auto-assigned color as an accent — buttons, highlights, text accents — 
on that page only. Must NOT leak into the global site theme, navigation, 
or any other item's page.
Note: Individual tasks/checklist items are too small to need page-level 
theming — 3.2 applies to subjects and folders only, not tasks.

3.3 REDESIGNED "ADD/EDIT" WINDOW (all items)
Fix: Fully redesign the modal used to add or edit any item type:
   - Clean, modern layout appropriate for a student tool.
   - Includes the "X" close button (3.4).
   - Fields adapt sensibly per item type (a subject's fields differ from 
     a checklist task's fields).
   - Visually consistent with the site's existing palette — this is a 
     component redesign, not a theme change.
   - Same window reused for both Add and Edit (pre-filled on Edit).
Acceptance: Fully usable on mobile with no field cut-off or overflow.

3.4 "X" CLOSE BUTTON ON ALL MODALS
Fix: Every modal/popup (Add/Edit window, confirmations, any overlay) gets 
a visible "X," typically top-right. Applies to modals/popups only — not 
full pages.

3.5 SUBJECT-ONLY SUB-STRUCTURE
Scope: This entire section applies to SUBJECTS ONLY — not folders, not 
tasks. Folders and tasks keep their current structure aside from Phases 
1–3's shared fixes above.
Fix: Each subject includes:
   a) A Google Drive folder link — for accessing that subject's lecture 
      files. One link per subject, editable, opens externally when tapped.
   b) A "Details" sub-page showing:
      - Progress (calculated — default formula: finished exams ÷ total 
        exams, expressed as %, unless a better metric is obvious from 
        existing data).
      - Number of lectures (manually entered by the user, editable).
      - Number of exams (manually entered by the user, editable).
      - Number of finished exams (tracked automatically by the site, 
        not manually entered).
   c) An "Exams" sub-page — list of exams tied to this subject.
   d) An "Attachments" sub-page for Telegram links. Each attachment 
      requires three tags before it can be saved:
      - Type: Study Sheet / Exam / Degree
      - Format: File / Image
      - Priority: Important / Not Important
      Use simple selectable controls (button groups/dropdowns) inside 
      the redesigned Add window style (3.3).
Acceptance: Every subject has all four sections; Details numbers update 
correctly as lectures/exams are added or marked finished; attachments 
cannot save without all three tags; folders and tasks are unaffected by 
this section.

3.6 NAVIGATION & BUTTON "LIQUID" MOTION STYLE
Fix:
   - Nav bar and buttons use a liquid/morphing animation only while 
     actively in motion (transitioning, dragging, mid-tap).
   - At rest, they're solid and crisp — no liquid effect when idle.
   - Every button plays a small liquid/morph feedback animation on 
     click/tap without changing position or size — visual feedback only.
Acceptance: Nothing shifts position permanently after a click animation 
completes.

════════════════════════════════════════
PHASE 4 — CALENDAR SYSTEM (BUILD & TEST SEPARATELY)
════════════════════════════════════════
Highest-risk feature. Build and test in isolation after Phases 1–3 are 
confirmed stable on the live site.

4.1 DAY / MONTH / YEAR VIEW TOGGLE (Schedule page)
   - DAY VIEW (existing behavior): horizontal day strip, tap a day to see 
     its tasks.
   - MONTH VIEW: full calendar grid for the selected month, weekday 
     headers (Sun–Sat), correct blank cells for days outside the month. 
     Tapping a date opens Day View for that date.
   - YEAR VIEW: grid of 12 months for the selected year. Tapping a month 
     opens Month View for that month.

4.2 SHARED SLIDER SELECTOR (reuses existing day-strip slider design)
   - Year selector: 50 entries starting from 1990 (1990–2039).
   - Month selector: 12 entries starting from January.
   - Day selector (within a specific month): dynamic count matching that 
     month's actual days (28–31), correctly handling leap years.

4.3 CURRENT DATE HIGHLIGHTING RULES
   - Year Grid: current year gets a distinct highlight.
   - Month Grid: current month gets a distinct highlight, only when 
     viewing its actual year.
   - Day Grid: current day gets a distinct highlight, only when viewing 
     the actual current month of the actual current year.

4.4 NAVIGATION FLOW
   - Year → tap month → Month View. Month → tap day → Day View (shows 
     that day's tasks).
   - Provide a clear way back up a level (breadcrumb, back arrow, or 
     tapping the current view's label).

Acceptance: Accurate calendar math (correct days/month, leap years); 
smooth transitions with no lost state; correct highlighting per 4.3; 
visually consistent with existing site design.
